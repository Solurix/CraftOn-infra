# 10 — Repository Strategy

We use **multiple repos** (not a monorepo), per project decision
(`adr/0003-multi-repo.md`).

## Repos

| Repo | Purpose | Status | Main tech |
|---|---|---|---|
| **`crafton`** (this one) | Infra (Terraform), docs, governance, project source-of-truth | ✅ exists | Terraform, Markdown |
| **`crafton-api`** | Backend API | ✅ exists | Python, FastAPI, SQLAlchemy, Alembic, pytest |
| **`crafton-web`** | Web/PWA frontend (workers + contractors + admin) | ✅ exists | Next.js, TypeScript, React |
| **`crafton-mobile`** | Native app (Flutter) — only if/when PWA limits require it | 🅿️ later | Flutter/Dart |

> Admin UI: Phase 1 lives inside `crafton-web` (a gated `/admin` area) to avoid a 4th repo
> early. Split out later only if it grows.

## What goes where
- **DB schema & migrations:** `crafton-api` (Alembic). The narrative model is documented
  here in `docs/05-data-model.md` — keep them in sync.
- **API contract:** authoritative OpenAPI is generated by `crafton-api`; the human
  overview is `docs/06-api-contract.md`.
- **Infra/Terraform:** here, in `infra/terraform/`.
- **Config/flag definitions:** documented here (`docs/07-config-and-flags.md`);
  implemented in `crafton-api`.
- **Cross-repo decisions:** here, as ADRs (`docs/adr/`).

## Cross-repo coordination (because multi-repo)
- This repo is the **hub**. Every repo's README links back here.
- Each app repo includes a short `CLAUDE.md` that says: "read the `crafton` repo docs
  first" and pins the key decisions.
- API changes that affect the web client: update `docs/06-api-contract.md` here in the
  same PR cycle and bump the API version if breaking.
- Shared types: the web app generates a typed client from the API's `/openapi.json`
  (no hand-maintained duplicate types).

## Conventions across all repos
- Default branch: `main`. (Currently pushing directly to `main` per owner instruction;
  introduce PR-based flow + branch protection when the team grows.)
- Conventional, descriptive commit messages.
- CI per repo (lint, type-check, tests) — see `09-testing-strategy.md`.
- Secrets via GCP Secret Manager; `.env.example` committed, real `.env` gitignored.
- Each repo has a `README.md` and a `CLAUDE.md`.

## Bootstrapping a new app repo (when created)
1. Add `README.md` + `CLAUDE.md` (link back here; pin decisions).
2. Add CI (lint/type/test) and `.env.example`.
3. Scaffold per `docs/04-phase-1-spec.md` build order.
4. Wire to infra outputs (DB connection, Storage bucket, Firebase project) from
   `infra/terraform/` outputs.
5. Update `docs/STATUS.md` here.
