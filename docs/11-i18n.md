# 11 — Internationalization (i18n) & Localization

## Requirement (locked)

- **App default language: Japanese (`ja`).** The product is Japanese-first; a new user
  with no preference sees Japanese.
- **Full English (`en`) translation.** English is a complete, first-class locale — every
  user-facing string has both `ja` and `en`. Not partial, not "added later."
- **Development language stays English.** Code identifiers, comments, docs, commit
  messages, and the **message keys** are English.

This is enforced from **Phase 1**, not deferred. See `adr/0008-i18n-ja-default-en-full.md`.

## The key idea (how the two reconcile)

Message **keys** are English identifiers; the **rendered default locale** is Japanese.

```
key:            "job.post.title"        ← English (dev language)
messages/ja.json: { "job.post.title": "求人を投稿" }   ← shown by default
messages/en.json: { "job.post.title": "Post a job" }   ← full parity required
```

So engineers read/write English everywhere in code, while end users get Japanese by
default and English on demand.

## Locale selection & fallback
- Default locale: **`ja`**.
- A logged-in user's `users.preferred_language` overrides the default and persists.
- Anonymous users: detect from `Accept-Language`, fall back to `ja`.
- Fallback for a missing key: **`ja`** (the source-of-truth locale). With the parity check
  below, missing keys should never reach production anyway.

## Scope — what gets translated
1. **Frontend UI copy** (labels, buttons, validation, empty/error states, onboarding).
2. **Backend-originated user-facing text:** API error messages, push/SMS/email templates,
   PDF/HTML terms documents, notification copy.
3. **Enumerated domain values shown to users** (e.g. status labels, tags).

**Not translated (kept as written):** user-generated content — chat messages, review
comments, profile free-text. (Dynamic *machine* translation of supervisor instructions to
a worker's language is a **separate** feature — see "Not the same as machine translation".)

## Not the same as machine translation
- **i18n (this doc):** static, human-authored app copy in `ja` + `en` (+ later `vi`, `id`).
  Phase 1.
- **Machine translation (`translation_enabled`, Cloud Translation):** dynamically
  translating *supervisor instructions* into a worker's language with construction terms.
  Phase 3. Different mechanism, different doc (`02-architecture.md`, `03-roadmap.md`).

Don't conflate them.

## Recommended implementation
- **Frontend (`crafton-web`, Next.js App Router):** `next-intl` with message catalogs
  `messages/ja.json` (source) and `messages/en.json`. `ja` as `defaultLocale`. No
  hardcoded user-facing strings in components.
- **Backend (`crafton-api`, FastAPI):** a small message-catalog layer (e.g. per-locale
  JSON/`gettext`) for error messages and notification/SMS/email templates, keyed the same
  way; choose locale from the authenticated user's `preferred_language` (or request
  `Accept-Language`).
- **Formatting:** locale-aware dates/numbers; **times display in Asia/Tokyo** regardless of
  locale; money formatted as JPY (integer yen).

## Completeness enforcement (so "full" stays true)
- **CI parity check:** `ja` and `en` catalogs must have the **identical key set** — build
  fails on any key present in one but missing/empty in the other. Applies to both frontend
  and backend catalogs. See `09-testing-strategy.md`.
- **No hardcoded strings:** lint/review rule — user-facing text must come from the catalog.
- Adding a string = add it to **both** `ja` and `en` in the same change.

## Future languages
- Vietnamese (`vi`), Indonesian (`id`) added by dropping in new catalogs, alongside the
  Phase 3 foreign-worker features. `supported_languages` config gates which are active
  (`07-config-and-flags.md`). These need not be 100% complete to ship the way `en` must —
  decide per-rollout, but `ja` and `en` are the always-complete pair.
